# Local lakeFS + MinIO (native binaries, no Docker)

Versioned private storage for evaluation datasets. MinIO holds the bytes
(`data/minio/`), lakeFS provides commits/tags on top (`data/lakefs/`). Both run
as plain processes — nothing is installed system-wide.

## Usage

```bash
./setup.sh    # one-time: download binaries, generate credentials, write config
./start.sh    # start both services (idempotent; safe to re-run)
./stop.sh     # stop both (data is preserved)
```

After setup, copy the printed credentials into the project `.env`:

```
LAKEFS_ENDPOINT=http://127.0.0.1:8000
LAKEFS_ACCESS_KEY_ID=...        # from local.env
LAKEFS_SECRET_ACCESS_KEY=...    # from local.env
```

## Layout

| Path          | Contents                                      | Committed? |
|---------------|-----------------------------------------------|------------|
| `bin/`        | minio, mc, lakefs, lakectl binaries           | no         |
| `data/minio/` | object bytes (lakeFS blockstore, bucket `lakefs-data`) | no |
| `data/lakefs/`| lakeFS metadata (local KV store)              | no         |
| `logs/`       | service logs                                  | no         |
| `local.env`   | generated MinIO/lakeFS credentials            | no         |
| `lakefs.yaml` | lakeFS config (regenerated by setup.sh)       | no         |
| `*.sh`        | setup/start/stop scripts                      | yes        |

## Notes

- lakeFS web UI: http://127.0.0.1:8000 (log in with the keys in `local.env`);
  MinIO console: http://127.0.0.1:9001.
- The lakeFS KV store is `local` (no Postgres) — fine for a single-node POC.
- Moving to a server later: run the same two services via docker-compose and
  change only `LAKEFS_ENDPOINT` in `.env`; no app-code changes.
- **Back up `data/` and `local.env`** if the datasets stored here matter — this
  machine is the only copy (unlike HF-hosted public data).
